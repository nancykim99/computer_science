# 💻 운영체제 (Operating System)

---

## 1. 운영체제와 컴퓨터

### 운영체제의 역할

- CPU, 메모리, I/O 장치 등 **하드웨어 자원을 관리**
- 프로세스·메모리·파일 시스템 관리
- 사용자와 하드웨어 사이의 **인터페이스** 역할

---

## 2. 메모리

### 메모리 계층

```
레지스터 > 캐시 > RAM > 디스크
(빠름/비쌈)          (느림/쌈)
```

- 위로 갈수록 속도 빠름, 용량 작음, 가격 비쌈

---

## 3. 프로세스와 스레드

### 프로세스 vs 스레드

| | 프로세스 | 스레드 |
|---|---|---|
| 정의 | 실행 중인 프로그램 | 프로세스 내의 실행 흐름 |
| 메모리 | 독립적 메모리 공간 | 코드·데이터·힙 공유 |
| 통신 | IPC 필요 (느리고 복잡) | 힙 통해 바로 공유 (빠르고 간단) |
| 컨텍스트 스위칭 | 느림 (캐시 비워야 함) | 빠름 (캐시 유지) |
| 장애 영향 | 독립적 (하나 죽어도 다른 것 무관) | 하나 죽으면 전체 프로세스 위험 |
| 동기화 | 불필요 | 필요 (공유 자원 때문) |

### 프로세스 메모리 구조

```
[높은 주소]
┌─────────────┐
│    스택     │  ← 함수 호출 시: 지역변수, 매개변수, 복귀 주소 저장 (↓ 방향으로 자람)
├─────────────┤
│      ↕      │  (스택 ↓, 힙 ↑)
├─────────────┤
│     힙      │  ← 동적 할당 메모리 (new, malloc) (↑ 방향으로 자람)
├─────────────┤
│   데이터    │  ← 전역 변수, 정적 변수 (static int x = 1)
├─────────────┤
│    코드     │  ← 실행 프로그램 코드 (읽기 전용)
└─────────────┘
[낮은 주소]
```

> **스레드 공유/독립**: 코드·데이터·힙은 공유 / 스택은 독립 (함수 호출 흐름이 다르기 때문)

### 스택 오버플로우

- 재귀 함수가 무한히 호출 → 복귀 주소가 스택에 계속 쌓임 → 스택 공간 초과

### 복귀 주소란?

- 함수 호출이 끝난 뒤 **어디로 돌아갈지** 저장해둔 주소
- `foo()` 호출 전, 다음 실행할 위치(주소)를 스택에 저장함

### 컨텍스트 스위칭

- CPU는 한 번에 하나의 프로세스만 실행 가능
- 여러 프로세스를 빠르게 번갈아 실행하여 **동시에 실행되는 것처럼** 보이게 함

#### 동작 방식

1. 프로세스 A 실행 중 → 인터럽트 발생
2. A의 상태(레지스터, PC 등)를 **PCB에 저장**
3. 다음 실행할 프로세스 B의 상태를 PCB에서 복원
4. 프로세스 B 실행 시작

> **오버헤드**: 스위칭 중 CPU가 아무 일도 하지 않음 → 너무 자주 발생하면 성능 저하

### PCB (Process Control Block)

- 운영체제가 프로세스를 관리하기 위한 **정보 저장 구조체**
- 프로세스 ID, 상태, 레지스터, PC 값 등 저장

### 멀티프로세스 vs 멀티스레드

| | 멀티프로세스 | 멀티스레드 |
|---|---|---|
| 방식 | 여러 독립 프로세스 | 하나의 프로세스 내 여러 스레드 |
| 메모리 | 독립 메모리 | 공유 메모리 |
| 통신 | IPC (복잡, 느림) | 힙 공유 (간단, 빠름) |
| 컨텍스트 스위칭 | 느림 | 빠름 |
| 안정성 | 높음 (독립적) | 낮음 (하나 죽으면 전체 위험) |
| 동기화 | 불필요 | 필요 |

---

### 공유 자원과 임계 영역

- **공유 자원**: 여러 스레드가 함께 접근할 수 있는 자원 (전역 변수, 파일, DB 등)
- **임계 영역(Critical Section)**: 공유 자원에 접근하는 코드 블록
  - 여러 스레드가 동시에 접근하면 **레이스 컨디션** 발생

#### 문제 해결 조건 3가지

1. **상호 배제**: 한 스레드가 임계 영역에 있으면 다른 스레드 진입 불가
2. **진행**: 임계 영역이 비어있으면 대기 중인 스레드가 진입 가능해야 함
3. **유한 대기**: 특정 스레드가 무한정 기다리는 일 없어야 함 (기아 상태 방지)

### 뮤텍스 vs 세마포어

| | 뮤텍스 | 세마포어 |
|---|---|---|
| 접근 수 | 1개 스레드만 | 여러 개 허용 가능 |
| 해제 주체 | 획득한 스레드만 | 다른 스레드도 가능 |
| 동작 | 락 획득/반환 | 카운터 감소/증가 |
| 활용 | 단일 자원 보호 | 제한된 수의 자원 관리 |

## 4. 교착 상태 (Deadlock)

### 정의

두 개 이상의 프로세스가 **서로 상대방이 가진 자원을 기다리며 영원히 대기**하는 상태

### 발생 조건 4가지 (모두 동시에 충족되어야 발생)

1. **상호 배제**: 한 번에 하나의 프로세스만 자원 사용 가능
2. **점유 대기**: 자원을 가진 채로 다른 자원을 기다림
3. **비선점**: 다른 프로세스의 자원을 강제로 빼앗을 수 없음
4. **순환 대기**: 프로세스들이 순환 형태로 서로 자원을 기다림 (A→B→A)

### 해결 방법 4가지

| 방법 | 설명 |
|---|---|
| **예방** | 발생 조건 4가지 중 하나를 원천 차단 |
| **회피** | 교착 상태 가능성 있으면 미리 피함 (은행원 알고리즘) |
| **회복** | 발생 후 프로세스 종료 또는 자원 강제 회수 |
| **무시** | 드물게 발생 시 그냥 무시 (실무에서 타임아웃과 함께 사용) |

> **은행원 알고리즘**: 자원 할당 전 안전 상태인지 검사 → 안전하면 할당, 아니면 대기

## 5. CPU 스케줄링 알고리즘

### 비선점형 방식

- 실행 중인 프로세스가 자발적으로 CPU를 내놓을 때까지 기다림

| 알고리즘 | 설명 |
|---|---|
| FCFS | 먼저 온 프로세스 먼저 처리 (선착순) |
| SJF | 실행 시간이 짧은 것 먼저 처리 |

### 선점형 방식

- 운영체제가 강제로 CPU를 회수할 수 있음

| 알고리즘 | 설명 |
|---|---|
| **Round Robin** | 동일한 time quantum 할당 후 순서대로 실행 |
| SRTF | 남은 실행 시간이 가장 짧은 프로세스 먼저 |
| 다단계 큐 | 우선순위별 큐로 나누어 처리 |

### Round Robin 상세

- 각 프로세스에 동일한 **time quantum** 할당 → 순서대로 돌아가며 실행
- 할당 시간 끝나면 다음 프로세스로 넘어가고 현재 프로세스는 맨 뒤로 이동
- 모든 프로세스가 공평하게 CPU 사용 가능
- n개 프로세스가 있으면 최대 `(n-1) × time quantum` 만큼만 기다리면 됨

#### Time Quantum 크기 영향

| 상황 | 결과 |
|---|---|
| 적당할 때 | 모든 프로세스가 골고루 CPU 사용, 응답 시간 빠름 |
| 너무 클 때 | FCFS(선착순)와 동일해짐 → CPU 독점 |
| 너무 작을 때 | 컨텍스트 스위칭 과다 발생 → 오버헤드 증가, 성능 저하 |

## 6. 동기/비동기 vs 블로킹/논블로킹

| | 동기 | 비동기 |
|---|---|---|
| 정의 | 작업이 끝날 때까지 기다린 후 다음 실행 | 작업을 시작해놓고 기다리지 않고 다음 실행 |
| 예시 | 전화 통화 (실시간으로 주고받음) | 문자 메시지 (보내놓고 다른 일) |

| | 블로킹 | 논블로킹 |
|---|---|---|
| 정의 | 제어권을 넘겨줌 → 다른 작업 못함 | 제어권을 바로 돌려받음 → 다른 작업 가능 |
| 예시 | 커피 주문 후 카운터 앞에서 대기 | 커피 주문 후 진동벨 받고 자리로 이동 |

## 💬 면접 예상 질문

**Q. 프로세스와 스레드의 차이는?**
> 프로세스는 독립적인 메모리 공간을 갖는 실행 단위. 스레드는 프로세스 내 실행 흐름으로 코드·데이터·힙을 공유하고 스택만 독립적으로 가짐.

**Q. 컨텍스트 스위칭이란?**
> CPU가 실행 중인 프로세스를 바꿀 때 현재 상태를 PCB에 저장하고 다음 프로세스 상태를 복원하는 과정. 스레드가 더 빠름 (캐시 유지).

**Q. 교착 상태 발생 조건 4가지는?**
> 상호 배제, 점유 대기, 비선점, 순환 대기. 4가지 모두 충족 시 발생.

**Q. Round Robin에서 time quantum이 너무 작으면?**
> 컨텍스트 스위칭이 과다하게 발생하여 오버헤드가 증가하고 성능이 저하됨.

**Q. 뮤텍스와 세마포어의 차이는?**
> 뮤텍스는 1개 스레드만 진입 가능하고 획득한 스레드만 해제 가능. 세마포어는 카운터 기반으로 여러 개 허용하며 다른 스레드도 해제 가능.
