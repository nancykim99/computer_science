# 🗄️ 데이터베이스 (Database)

---

## 1. 데이터베이스 기본

### 관계형 DB vs NoSQL

| | 관계형 DB (RDBMS) | NoSQL |
|---|---|---|
| 저장 방식 | 테이블 (행·열) | 다양한 형태 |
| 스키마 | 고정 스키마 | 유연한 스키마 |
| 확장 방식 | 수직 확장 | 수평 확장 |
| ACID | 보장 | 기본 미보장 |
| 쿼리 | SQL | 각자 다름 |
| 예시 | MySQL, PostgreSQL | MongoDB, Redis, Cassandra |

#### 관계형 DB 사용 시점
- 데이터 구조가 명확할 때
- 트랜잭션이 중요할 때 (은행, 결제)
- 복잡한 쿼리가 필요할 때

#### NoSQL 사용 시점
- 데이터 구조가 자주 바뀔 때
- 대용량 데이터 처리할 때 (SNS, 로그)
- 빠른 읽기/쓰기가 필요할 때
- 수평 확장이 필요할 때

#### NoSQL 종류

| 종류 | 저장 방식 | 예시 |
|---|---|---|
| Key-Value | 키:값 쌍 | Redis |
| Document | JSON 형태 문서 | MongoDB |
| Column | 열 중심 | Cassandra |
| Graph | 노드-엣지 관계 | Neo4j |

---

### 기본키 vs 외래키

**기본키 (Primary Key)**
- 테이블에서 각 행을 **유일하게 식별**하는 키
- 조건: **유일성** (중복 불가) / **최소성** (필요한 최소 컬럼만) / **NOT NULL** (반드시 값 존재)

**외래키 (Foreign Key)**
- 다른 테이블의 기본키를 **참조**하는 키
- 반드시 부모 테이블의 PK를 참조해야 함
- 참조 무결성 보장

## 2. ERD와 정규화

### 정규화

**정의**: 데이터 중복을 제거하고 테이블을 올바르게 분리하는 과정 → **이상 현상 방지**

#### 이상 현상 3가지

| 이상 현상 | 설명 |
|---|---|
| 삽입 이상 | 원하는 데이터만 추가하고 싶은데 다른 컬럼 값이 없어서 저장 불가 |
| 수정 이상 | 값을 수정하고 싶을 때 그 값을 가진 모든 행을 다 수정해야 함 |
| 삭제 이상 | 하나의 행을 삭제했더니 다른 필요한 정보도 함께 사라짐 |

#### 정규화 단계

| 단계 | 조건 |
|---|---|
| 1NF | 각 컬럼에 **원자값**(하나의 값)만 저장. 한 셀에 여러 값 금지 |
| 2NF | **부분 함수 종속 제거**: 기본키 일부에만 종속된 컬럼 분리 |
| 3NF | **이행 함수 종속 제거**: A→B→C 관계에서 별도 테이블로 분리 |

## 3. 트랜잭션과 무결성

### 트랜잭션

**정의**: 데이터베이스의 **논리적 작업 단위** — 여러 작업을 하나로 묶어 전부 성공하거나 전부 실패

> 예시: 계좌 이체 = 출금 + 입금 → 둘 다 성공하거나 둘 다 실패해야 함

### ACID

| 속성 | 설명 |
|---|---|
| **A (Atomicity)** 원자성 | 전부 성공하거나 전부 실패. All or Nothing. |
| **C (Consistency)** 일관성 | 트랜잭션 전후 데이터 일관성 유지 (DB 제약조건 항상 만족) |
| **I (Isolation)** 격리성 | 트랜잭션끼리 서로 간섭 불가 |
| **D (Durability)** 지속성 | 완료 후 영구 저장. 서버가 꺼져도 기록 유지. |

### 트랜잭션 격리 수준 (Isolation Level)

격리 수준이 낮을수록 → 서로 많이 볼 수 있음 → **성능 좋음**
격리 수준이 높을수록 → 서로 못 봄 → **데이터 정확함**

| 격리 수준 | 설명 | 발생 문제 |
|---|---|---|
| **READ UNCOMMITTED** | 커밋되지 않은 데이터도 읽기 가능 | Dirty Read |
| **READ COMMITTED** | 커밋된 데이터만 읽기 가능 | Non-repeatable Read |
| **REPEATABLE READ** | 반복 조회 시 항상 같은 값 보장 (MySQL InnoDB 기본값) | Phantom Read |
| **SERIALIZABLE** | 완벽한 격리, 순차 실행처럼 동작 | 없음 (성능 매우 저하) |

#### 격리 이상 현상

- **Dirty Read**: 커밋 안 된 데이터를 읽음 → T1이 롤백하면 T2는 존재하지 않는 값을 읽은 것
- **Non-repeatable Read**: 같은 행을 두 번 읽었을 때 결과가 다름 (중간에 다른 트랜잭션이 수정·커밋)
- **Phantom Read**: 같은 쿼리를 두 번 실행했을 때 결과 행 수가 다름 (중간에 INSERT 발생)

### Lock

**정의**: 여러 트랜잭션이 동시에 같은 데이터에 접근할 때 순서를 정해주는 것

| 종류 | 설명 |
|---|---|
| **공유 락 (Shared Lock)** | 읽기 락. 여러 트랜잭션 동시 읽기 가능. 읽는 중 수정 불가. |
| **베타 락 (Exclusive Lock)** | 쓰기 락. 한 트랜잭션만 수정 가능. 다른 트랜잭션 읽기도 불가. |

> Lock vs 트랜잭션: Lock은 **동시성 제어**, 트랜잭션은 **데이터 정합성 보장**. 트랜잭션이 내부적으로 Lock을 사용.

### 데드락 (Deadlock)

**정의**: 두 트랜잭션이 서로의 자원을 기다리며 영원히 끝나지 않는 상태

#### 해결 방법

| 방법 | 설명 |
|---|---|
| 예방 | 발생 조건 4가지 중 하나 제거 |
| 회피 | 은행원 알고리즘 — 안전 상태일 때만 자원 할당 |
| 탐지 후 복구 | 주기적으로 탐지 → 발견 시 트랜잭션 강제 롤백 |
| **타임아웃** | 일정 시간 대기 후 자동 취소 → **실무에서 가장 많이 사용** |

## 4. 데이터베이스 종류

### Redis

**정의**: 인메모리 기반의 Key-Value 구조를 사용하는 NoSQL 데이터베이스

| 항목 | 설명 |
|---|---|
| 저장 위치 | RAM (메모리) |
| 속도 | 매우 빠름 |
| 지원 자료구조 | List, Set, Hash, Sorted Set |
| 주 사용 목적 | 캐시 서버, 세션 저장소, 실시간 처리, 메시지 브로커 |
| 스레드 | 싱글 스레드 + 비동기 I/O |

#### 영속성 (Persistence)

원래 인메모리는 전원 끄면 데이터 사라짐 → RDB 스냅샷·AOF로 해결

| | RDB 스냅샷 | AOF (Append Only File) |
|---|---|---|
| 방식 | 특정 시점 메모리를 통째로 디스크에 저장 | 변경 명령어를 발생할 때마다 로그에 기록 |
| 데이터 손실 | 스냅샷 사이 구간 유실 | 거의 없음 |
| 파일 크기 | 작음 | 커질 수 있음 |
| 복구 속도 | 빠름 | 느림 |

#### Redis vs MySQL 비교

| 항목 | Redis | MySQL |
|---|---|---|
| 분류 | NoSQL: Key-Value | 관계형 DB |
| 저장 위치 | 메모리 | 디스크 |
| 속도 | 매우 빠름 | 상대적으로 느림 |
| 영속성 | 기본 없음 (설정 필요) | 기본 보장 |
| JOIN | 불가 | 가능 |

> 실무에서는 함께 사용: MySQL은 **원본 데이터 저장소**, Redis는 **캐시 계층**

```
사용자 요청 → Redis 캐시 확인
  → 있으면: 바로 반환 (빠름)
  → 없으면: MySQL 조회 후 Redis에 저장
```

## 5. 인덱스

**정의**: 데이터 위치를 저장해두는 자료구조 → 검색 속도 향상

| 장점 | 단점 |
|---|---|
| SELECT 검색 속도 향상: O(n) → O(log n) | 추가 저장 공간 필요 |
| | 삽입/수정/삭제 시 인덱스도 업데이트 → 쓰기 성능 저하 |

### B-트리 vs B+트리

| | B-트리 | B+트리 |
|---|---|---|
| 데이터 저장 | 모든 노드에 저장 | 리프 노드에만 저장 |
| 리프 노드 연결 | 없음 | 연결 리스트로 연결 |
| 범위 검색 | 가능 | 더 빠름 |

> 인덱스는 내부적으로 **B+트리** 사용

#### 해시 vs B+트리 비교

| | 해시 | B+트리 |
|---|---|---|
| 탐색 | O(1) | O(log n) |
| 범위 검색 | 불가 | 가능 |
| 인덱스 사용 | ❌ | ✅ |

## 6. 조인의 종류

**정의**: 두 개 이상의 테이블을 연결해서 하나의 결과로 합치는 연산

| 조인 종류 | 설명 |
|---|---|
| **INNER JOIN** | 두 테이블 모두에 존재하는 데이터만 반환 (교집합) |
| **LEFT JOIN** | 왼쪽 테이블 전부 + 오른쪽은 매칭된 것만 (없으면 NULL) |
| **RIGHT JOIN** | 오른쪽 테이블 전부 + 왼쪽은 매칭된 것만 (없으면 NULL) |
| **FULL OUTER JOIN** | 양쪽 테이블 모두 반환 (합집합, 없으면 NULL) |

---

## 7. 조인의 원리

| 원리 | 설명 | 복잡도 |
|---|---|---|
| **중첩 루프 조인** | 이중 for문처럼 하나씩 비교 | O(n × m) |
| **정렬 병합 조인** | 두 테이블 정렬 후 병합 | O(n log n) |
| **해시 조인** | 해시 테이블 만들어서 매칭 | O(n + m) |

## 💬 면접 예상 질문

**Q. 트랜잭션 ACID란?**
> 원자성(All or Nothing), 일관성(규칙 항상 만족), 격리성(트랜잭션 간 간섭 없음), 지속성(영구 저장).

**Q. 트랜잭션 격리 수준 4단계는?**
> READ UNCOMMITTED → READ COMMITTED → REPEATABLE READ → SERIALIZABLE. 위로 갈수록 성능 저하, 데이터 정확도 향상.

**Q. 인덱스의 장단점은?**
> 검색 성능 향상(O(n)→O(log n))이지만, 추가 저장 공간 필요하고 쓰기 성능 저하.

**Q. INNER JOIN과 LEFT JOIN의 차이는?**
> INNER JOIN은 양쪽 모두 존재하는 데이터만 반환. LEFT JOIN은 왼쪽 전부 반환하고 오른쪽은 매칭 없으면 NULL.

**Q. 관계형 DB와 NoSQL의 차이는?**
> 관계형 DB는 고정 스키마·ACID 보장·수직 확장. NoSQL은 유연한 스키마·수평 확장·대용량 처리에 적합.

**Q. Redis는 왜 빠른가?**
> 데이터를 디스크가 아닌 RAM에 저장하기 때문. 메모리 접근은 디스크보다 수백~수천 배 빠름.
