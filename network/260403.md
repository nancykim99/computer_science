# 🌐 네트워크 (Network)

---

## 1. 네트워크의 기초

### 처리량과 지연 시간

- **처리량 (Throughput)**: 단위 시간당 전송할 수 있는 데이터의 양 (bps)
- **지연 시간 (Latency)**: 데이터가 출발지에서 목적지까지 도달하는 데 걸리는 시간
  - 전파 지연, 처리 지연, 큐잉 지연, 전송 지연으로 구성

### 네트워크 토폴로지

네트워크를 구성하는 노드와 링크의 **물리적/논리적 배치 형태**

| 토폴로지 | 특징 | 장점 | 단점 |
|---|---|---|---|
| 버스형 | 하나의 회선에 모두 연결 | 설치 간단, 비용 저렴 | 중앙 회선 장애 시 전체 마비 |
| 스타형 | 중앙 허브에 모두 연결 | 장애 감지 쉬움 | 허브 고장 시 전체 마비 |
| 링형 | 원형으로 연결 | 구현 간단 | 하나 끊기면 전체 영향 |
| 메시형 | 모든 노드가 서로 연결 | 장애에 강함 | 비용 높음 |
| 트리형 | 계층적 구조 | 확장 용이 | 상위 노드 장애 시 하위 영향 |

**병목 현상**: 특정 구간에서 트래픽이 집중되어 성능이 저하되는 현상

### 네트워크 분류

| 분류 | 범위 | 설명 |
|---|---|---|
| LAN | 근거리 (건물/캠퍼스) | 빠른 속도, 낮은 오류율 |
| MAN | 도시 단위 | LAN과 WAN의 중간 |
| WAN | 국가/대륙 이상 | 인터넷, 느린 속도 |

### 네트워크 성능 분석 명령어

| 명령어 | 용도 |
|---|---|
| `ping` | 대상 호스트까지 도달 가능 여부, 응답 시간 측정 |
| `traceroute` (tracert) | 패킷이 거치는 경로 및 각 홉의 지연 시간 확인 |
| `netstat` | 네트워크 연결 상태, 포트 사용 현황 확인 |
| `nslookup` | DNS 조회, 도메인 → IP 변환 확인 |

## 2. TCP/IP 4계층 모델

### 계층 구조

| 계층 | 이름 | 역할 | 프로토콜/장비 |
|---|---|---|---|
| 4 | **애플리케이션 계층** | 사용자에게 서비스 제공 | HTTP, HTTPS, FTP, DNS, SMTP |
| 3 | **전송 계층** | 신뢰성 있는 데이터 전송, 포트 기반 | TCP, UDP |
| 2 | **인터넷 계층** | IP 주소 기반 라우팅 | IP, ARP, ICMP |
| 1 | **네트워크 접근 계층** | 물리적 전송, MAC 주소 기반 | 이더넷, Wi-Fi |

> TCP/IP 4계층은 OSI 7계층을 단순화한 모델

#### OSI 7계층 vs TCP/IP 4계층

| OSI 7계층 | TCP/IP 4계층 |
|---|---|
| 응용·표현·세션 계층 | 애플리케이션 계층 |
| 전송 계층 | 전송 계층 |
| 네트워크 계층 | 인터넷 계층 |
| 데이터 링크·물리 계층 | 네트워크 접근 계층 |

### TCP vs UDP

| | TCP | UDP |
|---|---|---|
| 연결 방식 | 연결형 (3-way handshake) | 비연결형 |
| 신뢰성 | 높음 (재전송 보장) | 낮음 (보장 없음) |
| 순서 보장 | O | X |
| 속도 | 느림 | 빠름 |
| 활용 | 파일 전송, HTTP, 이메일 | 스트리밍, 게임, DNS |

#### 3-way handshake (TCP 연결 수립)

```
Client → Server : SYN
Server → Client : SYN + ACK
Client → Server : ACK
→ 연결 수립 완료
```

#### 4-way handshake (TCP 연결 종료)

```
Client → Server : FIN
Server → Client : ACK
Server → Client : FIN
Client → Server : ACK
→ 연결 종료
```

### PDU (Protocol Data Unit)

각 계층에서 데이터가 어떤 단위로 전송되는지

| 계층 | PDU 이름 |
|---|---|
| 애플리케이션 | 메시지 (Message) |
| 전송 | 세그먼트 (Segment) — TCP / 데이터그램 — UDP |
| 인터넷 | 패킷 (Packet) |
| 네트워크 접근 | 프레임 (Frame) |

## 3. 네트워크 기기

### 계층별 처리 기기

| 계층 | 기기 | 역할 |
|---|---|---|
| 애플리케이션 | **로드 밸런서**, **방화벽** | 트래픽 분산, 접근 제어 |
| 인터넷 (네트워크) | **라우터** | IP 기반 최적 경로 결정, 서로 다른 네트워크 연결 |
| 데이터 링크 | **스위치** | MAC 주소 기반 같은 네트워크 내 통신 |
| 물리 | **허브**, **리피터** | 신호 증폭, 단순 전달 |

#### 라우터 vs 스위치 vs 허브

| | 허브 | 스위치 | 라우터 |
|---|---|---|---|
| 계층 | 1 (물리) | 2 (데이터 링크) | 3 (네트워크) |
| 주소 기반 | 없음 | MAC 주소 | IP 주소 |
| 전달 방식 | 모든 포트에 전달 | 특정 포트에 전달 | 최적 경로 결정 |

## 4. IP 주소

### ARP (Address Resolution Protocol)

- **IP 주소 → MAC 주소** 변환 프로토콜
- 같은 네트워크 내에서 상대방의 MAC 주소를 모를 때 브로드캐스트로 질의

```
"192.168.1.5 MAC 주소 가진 분 누구세요?" → 브로드캐스트
→ 해당 IP 가진 장치가 MAC 주소로 응답
```

### 홉바이홉 통신

- 패킷이 **라우터를 하나씩 거쳐** 목적지까지 도달하는 방식
- 각 라우터는 라우팅 테이블을 참고해 다음 경로 결정
- 한 번의 라우터 경유 = 1 hop

### IP 주소 체계

#### IPv4

- 32비트, 4개 옥텟 (예: 192.168.0.1)
- 약 43억 개 주소 (부족 문제 발생)

#### IPv6

- 128비트, 16진수 표현 (예: 2001:0db8:85a3::8a2e:0370:7334)
- 사실상 무한한 주소 공간

#### 클래스 기반 주소 체계

| 클래스 | 범위 | 용도 |
|---|---|---|
| A | 0.0.0.0 ~ 127.255.255.255 | 대규모 네트워크 |
| B | 128.0.0.0 ~ 191.255.255.255 | 중규모 네트워크 |
| C | 192.0.0.0 ~ 223.255.255.255 | 소규모 네트워크 |

#### 사설 IP vs 공인 IP

- **공인 IP**: 인터넷에서 직접 사용하는 고유 주소
- **사설 IP**: 내부 네트워크에서만 사용 (192.168.x.x, 10.x.x.x)
- **NAT**: 사설 IP → 공인 IP로 변환해주는 기술

## 5. HTTP

### HTTP/1.0

- 연결 당 하나의 요청/응답 처리 → 매번 새로운 TCP 연결
- 비효율적 (연결 오버헤드 큼)

### HTTP/1.1

- **Keep-Alive**: 하나의 TCP 연결로 여러 요청/응답 처리
- **파이프라이닝**: 요청을 연속으로 보낼 수 있음
- 단점: **HOL Blocking** — 앞 요청이 막히면 뒤 요청도 대기

### HTTP/2

- **멀티플렉싱**: 하나의 TCP 연결에서 여러 요청을 **동시에** 처리
- **헤더 압축**: HPACK 알고리즘으로 헤더 크기 축소
- **서버 푸시**: 요청 없이 서버가 먼저 리소스 전송
- HOL Blocking 해결 (애플리케이션 레벨)

### HTTPS

- HTTP + **TLS/SSL 암호화**
- **기밀성**: 데이터 암호화
- **무결성**: 데이터 변조 방지
- **인증**: 서버 신원 확인 (인증서)

#### TLS 핸드셰이크 과정

```
1. Client Hello — 지원하는 암호화 방식 목록 전송
2. Server Hello — 선택한 암호화 방식 + 인증서 전송
3. 인증서 검증 — CA(인증기관)를 통해 서버 신원 확인
4. 키 교환 — 세션 키(대칭 키) 생성
5. 암호화 통신 시작
```

### HTTP/3

- **QUIC 프로토콜** 기반 (TCP 대신 **UDP** 사용)
- TCP의 HOL Blocking 완전 해결
- 연결 수립 시간 단축 (0-RTT 가능)
- 모바일 환경에서 네트워크 전환 시 유리

### HTTP 버전 비교

| | HTTP/1.0 | HTTP/1.1 | HTTP/2 | HTTP/3 |
|---|---|---|---|---|
| 전송 프로토콜 | TCP | TCP | TCP | UDP (QUIC) |
| 연결 방식 | 매번 새 연결 | Keep-Alive | 멀티플렉싱 | 멀티플렉싱 |
| HOL Blocking | 있음 | 있음 | 부분 해결 | 완전 해결 |
| 헤더 압축 | X | X | O (HPACK) | O (QPACK) |

## 💬 면접 예상 질문

**Q. TCP와 UDP의 차이는?**
> TCP는 연결형 프로토콜로 신뢰성 보장 (재전송, 순서 보장). UDP는 비연결형으로 빠르지만 신뢰성 없음. 파일 전송은 TCP, 실시간 스트리밍은 UDP.

**Q. 3-way handshake란?**
> TCP 연결 수립 과정. Client→Server SYN, Server→Client SYN+ACK, Client→Server ACK의 3단계로 연결을 확립.

**Q. HTTP와 HTTPS의 차이는?**
> HTTPS는 HTTP에 TLS/SSL 암호화를 추가. 데이터 기밀성·무결성·서버 인증을 보장.

**Q. HTTP/2의 주요 개선점은?**
> 멀티플렉싱(동시 요청 처리), 헤더 압축(HPACK), 서버 푸시. HTTP/1.1의 HOL Blocking을 애플리케이션 레벨에서 해결.

**Q. ARP란?**
> IP 주소를 MAC 주소로 변환하는 프로토콜. 같은 네트워크 내에서 브로드캐스트로 MAC 주소를 질의함.

**Q. 라우터와 스위치의 차이는?**
> 스위치는 MAC 주소 기반으로 같은 네트워크 내 통신을 담당 (2계층). 라우터는 IP 주소 기반으로 서로 다른 네트워크 간 최적 경로를 결정 (3계층).
